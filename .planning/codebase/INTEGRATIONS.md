# External Integrations

**Analysis Date:** 2026-04-13

## APIs & External Services

**CIK Election Commission:**
- **Live count HTML pages** (during election night, ~20:30 onward):
  - `results.cik.bg/pe202410/rezultati/index.html` — National vote table (27 parties), updated every 60 seconds
  - `results.cik.bg/pe202410/rezultati/NN.html` (×32) — Per-МИР drill-down tables, updated every 5 minutes
  - `results.cik.bg/pe202410/aktivnost/index.html` — Turnout snapshots (11:00, 17:00, 20:00), polled every 5 minutes
  - `results.cik.bg/pe202410/map/index.html` — Inline SVG choropleth with МИР paths (geometry, pulled once and cached)
  - SDK/Client: `httpx` (async HTTP/2) + `selectolax`/`lxml` for parsing
  - Expected behavior: 404 at <14% counted; data becomes available after preliminary count near-complete
  - Regression testing: Wayback Machine snapshots at `web.archive.org/web/*/results.cik.bg/pe202410/...`

- **Post-decision pages** (T+2 to T+5 days after election):
  - `results.cik.bg/pe202410_ks/rezultati/index.html` — Final seat allocation added
  - `results.cik.bg/pe202410_ks/rezultati/NN.html` — Per-МИР final seats
  - `results.cik.bg/pe202410_ks/hnm.64.html` — Full Hare-Niemeyer methodology (130 tables with seat allocation algorithm)
  - `results.cik.bg/pe202410_ks/hnm/index.html` — Human-readable seat allocation explainer
  - Parser: Extract seat-per-МИР table for swing charts and live projection validation

**CIK Opendata Exports:**
- **URL pattern:** `results.cik.bg/pe{YYMMDD}/opendata/`
  - Availability: After preliminary count near-complete (not live at <14% counted)
  - Polling cadence during post-count: every 15 minutes until stable for 2 hours
  - Files: `export.zip` (CSV bundle), `spreadsheet.zip` (XLSX bundle)
  - Purpose: Source-of-truth vote data, used to validate HTML scraper numbers and populate final analysis
  - Checksum approach: ETag / Last-Modified headers before GET to avoid hammering

- **CSV structure (in export.zip):**
  - `cik_parties_27.10.2024.txt` — National parties (27 rows)
  - `local_parties_27.10.2024.txt` — Party registrations per МИР (840 rows)
  - `sections_27.10.2024.txt` — All polling stations (12,919 rows, with EKATTE codes)
  - `protocols_27.10.2024.txt` — SIK protocol headers (turnout, valid/invalid ballots)
  - `votes_27.10.2024.txt` — Per-station variable-length vote totals by party
  - `preferences_27.10.2024.txt` — Per-station × per-candidate preference votes (2,998,892 rows)
  - Delimiter: `;` (semicolon)
  - Encoding: UTF-8

- **XLSX structure (in spreadsheet.zip):**
  - 32 files: `ns01.xlsx` … `ns32.xlsx` (one per МИР)
  - Each file has 7 sheets: Секции, Състави, Кандидати, Точки в протоколи, Гласове, Преференции, Легенда
  - Human-readable election data at МИР granularity

- **Voting machine raw logs (suemg/):**
  - ZIP files under МИР subfolders: `suemg/NN/*.zip` (9,447 zips total)
  - Each ZIP contains: `NNNNNNNNN.csv` (per-machine vote log) + `NNNNNNNNN.csv.p7s` (PKCS#7 signature)
  - Purpose: Verify voting machine authenticity, cross-check against protocol totals
  - Signature validation: Check `.p7s` against `.csv` to prove unmodified machine output

## Data Storage

**Geometry Files (GeoJSON):**
- `bg.json` — 28 oblasti polygons
  - Source: simplemaps.com (attribution in footer of `map.html`)
  - Format: GeoJSON FeatureCollection
  - Properties per oblast: name, center coordinates
  - Usage: Choropleth polygons in SVG rendering

- `bg-municipalities.json` — 265 municipalities polygons
  - Source: yurukov/Bulgaria-geocoding GitHub repository
  - Format: GeoJSON FeatureCollection
  - Properties: `nuts4` (NUTS-4 code), `nuts3` (NUTS-3 code), `name` (municipality name)
  - URL pattern: `https://raw.githubusercontent.com/yurukov/Bulgaria-geocoding/master/municipalities_names.geojson`
  - Usage: Municipality-level choropleths

**Results Data (JSON):**
- `results.json` — National vote shares, oblast totals, processing progress
  - Structure: `{parties: [...], countyList: {...}, oblast_totals: {...}, grand_total: ..., lastUpdated: ..., processingPctDomestic: ...}`
  - Generated by Python aggregation script from `votes_27.10.2024.txt`
  - Real-time usage: Fetch with `?cache-buster` timestamp to bypass browser cache

- `results_municipalities.json` — Same data keyed by party number (1–28)
  - Structure: `{parties: [...], data: {party_num: {oblast_code: {votes: ..., pct: ...}}}, oblast_totals: {...}, grand_total: ...}`
  - Alternative format for client-side lookups

- `data_by_oblast.json` — Per-oblast summary (regional totals)
  - Generated from rolled-up `votes_27.10.2024.txt` by oblast EKATTE rollup

**File Storage:**
- Local filesystem only (no external CDN, S3, or blob storage)
- Co-hosted with HTML on same origin (GitHub Pages for production)

**Caching:**
- Browser cache disabled for JSON results (cache-busting: `fetch('results.json?' + Date.now())`)
- Static geometry files (bg.json, bg-municipalities.json) cached indefinitely

## External Geometry & Boundaries

**Data sources for geo assets:**

- **МИР (31 districts):** Extract inline SVG paths from `results.cik.bg/pe202410/map/index.html` (elements: `g#mip1` … `g#mip31`)

- **Области (28 regions):** 
  - Primary: simplemaps.com (already in `bg.json`)
  - Alternative: OSM `admin_level=4` via Overpass API

- **Общини (265 municipalities):**
  - Primary: yurukov/Bulgaria-geocoding (already in `bg-municipalities.json`)
  - Alternative: OSM `admin_level=6` via Overpass API

- **Населени места (~5,300 settlements):**
  - OSM `place=*` + EKATTE mapping (NSI publishes EKATTE↔OSM crosswalk)

## Data Comparison & Validation

**2024 vs 2026 comparisons require:**
- Manual party crosswalk (YAML lookup) to map 2024 parties ↔ 2026 parties/coalitions
- Swing charts compute: `(2026_votes - 2024_votes) / 2024_votes × 100%`
- Polling-station level swing has missing rows (some СИК added/split/merged between elections); aggregate to EKATTE or община for robustness
