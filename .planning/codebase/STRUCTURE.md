## Directory Purposes

**Project Root:**
- Purpose: Application entry point and data artefacts
- Contains: HTML visualizations, pre-computed JSON data files, documentation
- Key files: `map.html`, `map_municipalities.html`, `results.json`, `bg.json`

**`./export/` (Data Input – NOT Shipped):**
- Purpose: Archive of raw CIK opendata in machine-readable CSV + digital signatures
- Contains: 
  - `Актуализирана база данни/`: 8 CSVs with schemas, parties, candidates, polling stations, protocols, per-station votes, preferences, machine signatures (PKCS#7)
  - `suemg/`: 33 subdirectories (one per МИР), each with ~280 `.zip` files containing per-machine vote logs and signatures
- Size: 123 MB total
- Committed: No (not part of the app)
- Role: Source of truth for vote data; re-aggregated by external Python scripts to produce `/results*.json`

**`./spreadsheet/` (Data Input – NOT Shipped):**
- Purpose: Archive of raw CIK opendata in human-readable Excel format
- Contains: 32 `.xlsx` files (one per МИР), each with 7 sheets (polling stations, party registrations, candidates, protocol points, votes, preferences, legend)
- Size: 65 MB total
- Committed: No (not part of the app)
- Role: Human-readable reference; same underlying data as `./export/`

**`./.planning/codebase/` (Generated):**
- Purpose: GSD orchestrator metadata for code generation and planning
- Contains: `ARCHITECTURE.md`, `STRUCTURE.md`, `CONVENTIONS.md`, `TESTING.md`, `CONCERNS.md`
- Generated by: `/gsd:map-codebase` commands
- Consumed by: `/gsd:plan-phase` and `/gsd:execute-phase` commands

## Key File Locations

**Entry Points:**

- `map.html`: Oblast-level visualization. Entry point for `/oblast/` URL.
- `map_municipalities.html`: Municipality-level visualization. Entry point for `/municipalities/` URL.

**Geometry Data (GeoJSON):**

- `bg.json`: 28 oblasts with ISO 3166-2 codes (`BG01`–`BG28`). Properties: `id` (ISO code), `name` (English oblast name), `geometry` (Polygon coordinates).
- `bg-municipalities.json`: 265 municipalities with NUTS4 codes (e.g., `BLG01`, `SOF46`). Properties: `nuts4` (NUTS4 code), `nuts3` (country code `BLG`), `name` (Bulgarian municipality name). Note: Does NOT include `name_en` in GeoJSON; bilingual names come from `results_municipalities.json` via `name_en` field.

**Aggregated Results:**

- `results.json`: Vote shares by party per oblast. Structure: `{ parties: [...], countyList: { BG01: {...}, ... }, oblast_totals, grand_total, lastUpdated, processingPctDomestic }`. Used by `map.html`.
- `results_municipalities.json`: Vote shares by party per municipality. Structure: `{ parties: [...], municipalityList: { BLG01: {name, name_en, PARTY_KEY: percent, ...}, ... } }`. Used by `map_municipalities.html`. **Important:** Municipality results include `name_en` field (bilingual support).
- `data_by_oblast.json`: Optional aggregated statistics; currently not consumed by either visualization.

**Documentation:**

- `PLAN.md`: Technical specification, data schema, research notes on electoral systems, geographic boundaries, and project scope.

## Naming Conventions

**Files:**

- HTML visualizations: lowercase with underscores (`map.html`, `map_municipalities.html`)
- JSON data: lowercase with underscores (`results.json`, `results_municipalities.json`, `bg.json`, `bg-municipalities.json`, `data_by_oblast.json`)
- Markdown docs: uppercase (`PLAN.md`)

**Directories:**

- Raw data archives: singular, lowercase (`export/`, `spreadsheet/`)
- Project metadata: dot-prefixed (`/.planning/`)

**Geographic Identifiers (in JSON):**

- Oblasts: ISO 3166-2 codes `BG01`–`BG28` (28 regions)
- Municipalities: NUTS4 codes starting with country code, e.g., `BLG01`–`BLG52` (Blagoevgrad oblast), `SOF01`–`SOF46` (Sofia City)
- Internal CIK МИР numbers: `01`–`32` (used in raw data, not in app)
- NUTS3: Always `BLG` (Bulgaria)

## Where to Add New Code

**New Visualization Page (e.g., District-level Map):**
- Primary code: Create `/map_districts.html` by copying `map_municipalities.html` and modifying:
  - Fetch URL: Change `'bg-municipalities.json'` to `'bg-districts.json'` and `'results_municipalities.json'` to `'results_districts.json'`
  - CSS: Update title in `<h1>` and subtitle in `.subtitle`
  - JavaScript: Update labels to reference district codes instead of municipality codes
- Data: Generate corresponding `bg-districts.json` and `results_districts.json` via Python aggregation pipeline
- Tests: None (no test framework in place)

**New Legend Feature or Interactive Control:**
- Primary code: Modify `<script>` block in relevant HTML file (e.g., `map.html`)
- Pattern: Add state variable (e.g., `let hideZeroes = false`), button to toggle in legend, conditional rendering in `buildMap()` and `buildLegend()`
- No shared module: Duplicates must be made to both HTML files

**Shared Helper Functions:**
- Currently: No shared file. Code is duplicated (~1700 lines per HTML file, ~80% overlap).
- Recommendation: Consider extracting shared functions to a `/lib.js` file (colour helpers, geometry, tooltip logic) and loading with `<script src="lib.js"></script>` before the IIFE. This would reduce duplication but requires adding a build step or module loader.
- Until then: Maintain both copies in sync; any bug fix or enhancement must be applied to both HTML files.

**Utilities and Helpers:**
- Location: Inline in `<script>` block of relevant HTML (no separate `.js` files)
- Scope: Global functions (e.g., `hexToRgb`, `partyShade`, `makeProjection`) defined at module level
- Naming: camelCase, descriptive (e.g., `cssId`, `bbox`, `featurePath`, `getWinner`)

## Page-Specific Data Fields

**`map.html` (Oblasts):**
- Geometry keys: `BG01`–`BG28` (ISO codes)
- Result keys: Identical (`BG01`–`BG28`)
- Party fields: Numeric vote percentages (e.g., `GERB-SDS: 27.54`)
- Names: Hardcoded in `bg.json` properties (English only)

**`map_municipalities.html` (Municipalities):**
- Geometry keys: NUTS4 codes (e.g., `BLG01`, `SOF46`)
- Result keys: Identical NUTS4 codes
- Party fields: Numeric vote percentages
- Names: Bilingual via `name_en` field in `results_municipalities.json` (Bulgarian `name` + English `name_en`)
- Special note: Geometry GeoJSON does NOT include `name_en`; it comes from results JSON
